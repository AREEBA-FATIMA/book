## 1. High-Level Architecture

### Structure of the Docusaurus Site
The Docusaurus site will follow a standard structure, enhanced by Spec-Kit Plus for automated generation:
-   `/`: Root directory containing `docusaurus.config.ts`, `package.json`, and other configuration files.
-   `/src`: Contains custom components, CSS, and potentially pages that are not part of the book content.
-   `/docs`: This will be the primary content directory for the book, containing all MDX chapters and organized into subdirectories representing logical sections or chapters.
-   `/static`: For static assets like images.

### Chapter Layout
Chapters will be organized within the `/docs` directory. Each chapter will be an MDX file, potentially within its own subdirectory for better organization if sub-sections are required.

### File + Folder Generation Strategy
All files and folders will be generated by Claude Code based on detailed specifications.
1.  **Top-level configuration files**: `docusaurus.config.ts`, `package.json`, `tsconfig.json`, etc., will be generated from a `config-spec.md` or similar.
2.  **Content files**: MDX files for chapters and the landing page will be generated from individual `chapter-spec.md` files or a `book-content-spec.md`.
3.  **Asset files**: Any necessary images or other static assets will be referenced in the MDX and placed in the `/static` directory by generated code.
4.  **Sidebar configuration**: The `sidebars.js` (or `sidebars.ts`) file will be auto-generated based on the detected structure of the `/docs` directory and a `sidebar-spec.md`.

## 2. Component Breakdown (Specification Focus)

Each key component of the Docusaurus book will have a dedicated specification document that drives its AI generation.

### Landing Page Spec
-   **Purpose**: Defines the content, structure, and layout of the book's introductory landing page.
-   **Input**: `specs/project-1/landing-page-spec.md` (example path)
-   **Output**: `docs/landing-page.mdx`

### Chapters Specs
-   **Purpose**: Defines the content, structure, and hierarchy for each individual chapter of the book. Minimum of 5 chapters.
-   **Input**: `specs/project-1/chapter-1-spec.md`, `specs/project-1/chapter-2-spec.md`, etc.
-   **Output**: `docs/chapter-1/index.mdx`, `docs/chapter-2/index.mdx`, etc.

### Sidebar + Config Specs
-   **Purpose**: Defines the overall Docusaurus configuration (title, tagline, plugins, themes, presets) and the automatic sidebar generation logic.
-   **Input**: `specs/project-1/docusaurus-config-spec.md`, `specs/project-1/sidebar-config-spec.md`
-   **Output**: `docusaurus.config.ts`, `sidebars.js` (or `sidebars.ts`)

### GitHub Pages Deploy Spec
-   **Purpose**: Defines the GitHub Actions workflow or scripts necessary to build and deploy the Docusaurus site to GitHub Pages.
-   **Input**: `specs/project-1/deployment-spec.md`
-   **Output**: `.github/workflows/deploy.yml`

## 3. Execution Flow (Step-by-Step)

This section outlines the step-by-step process, leveraging Spec-Kit Plus and Claude Code for a 100% AI-driven workflow.

1.  **Write Specifications (Human/AI collaboration)**:
    *   **Action**: Create or refine all necessary `.spec` files (e.g., `landing-page-spec.md`, `chapter-x-spec.md`, `docusaurus-config-spec.md`, `deployment-spec.md`) within the `specs/project-1/` directory. These specifications will be the *only* source of truth for generation.
    *   **Responsibility**: Primarily human-driven definition, with AI assistance for drafting and validation.
    *   **Dependency**: Constitution.md must be finalized.

2.  **Run AI Generation (Claude Code)**:
    *   **Action**: Execute Claude Code (via Spec-Kit Plus commands) to process all `.spec` files and generate the Docusaurus project structure, configuration, and MDX content. This includes running initial Docusaurus setup commands if not already present.
    *   **Responsibility**: Claude Code.
    *   **Dependency**: All `.spec` files must be complete and valid.

3.  **Validate Outputs (Automated/AI)**:
    *   **Action**: Claude Code will perform checks to ensure generated files conform to the specifications and Docusaurus best practices. This includes syntax checks, link validation within MDX, and structural integrity.
    *   **Responsibility**: Claude Code (automated validation scripts and potentially AI-driven code review).
    *   **Dependency**: Code generation complete.

4.  **Build (Docusaurus)**:
    *   **Action**: Run `npm install` followed by `npm run build` within the generated Docusaurus project to compile the site.
    *   **Responsibility**: Claude Code (executing Bash commands).
    *   **Dependency**: Valid generated code and dependencies installed.

5.  **Deploy (GitHub Pages)**:
    *   **Action**: Execute the defined GitHub Actions workflow or deployment script to publish the built Docusaurus site to GitHub Pages.
    *   **Responsibility**: Claude Code (triggering deployment via Bash commands or Git operations).
    *   **Dependency**: Successful build.
